# dota v2.0.0 -- Fragmented Vault + Mediated Trust

## Context

v1.1.0 closed the open SECURITY-AUDIT.md findings and moved file operations
to FD-based syscalls with O_NOFOLLOW and post-fstat identity checks. That
release landed on `claude/fix-security-tests-kH8TP` through commit
`351f96f`.

The remaining concerns the user surfaced are not within the v1.1.0 audit
subspace -- they are architectural. v2.0.0 is the milestone for that work.
The four directions that shape it:

1. **The single-file vault is itself a weakness.** Anyone who can read
   `~/.dota/vault.json` has the full at-rest target. The v2 storage
   model is a set of small fragments where *no single fragment reveals
   anything on its own*, composed at unlock time via an ordering derived
   from a passphrase- and session-scoped chain.
2. **Conventional IPC transports (Unix domain sockets, systemd socket
   activation, D-Bus) are insufficient** against a same-UID attacker who
   can attach to the kernel buffers or race the socket file. v2's
   mediated-trust direction is hardware-backed (TEE / TPM / Secure
   Enclave), not user-mode IPC.
3. **UEFI is itself an attack surface in its current state** (LogoFAIL,
   BootHole, BlackLotus, ESPecter). The pre-boot daemon idea remains a
   research target, not a deliverable.
4. **The remaining code-level hardening from the v1.x audit follow-up**
   (ellipsis ban, allowlist serde, side-channel posture documentation,
   substantive `generate_salt` rationale) ships with v2.0 rather than
   accumulating into a separate v1.2 release.

Recorded user decisions:

- **Release shape**: v2.0.0 = single major milestone. Hardening + storage
  redesign + TPM integration + TEE/daemon research ride together.
- **Vault size**: fragmented storage with composition ordering ("door
  knock at REST"). Not a single-file MAX bound. See section 1.
- **UEFI**: research-grade, defer. Document the attack surface and the
  hardware partnership it implies; don't write a UEFI app yet.
- **IPC**: rejected -- UDS, systemd socket activation, and D-Bus are all
  too weak. Hardware-rooted mediation (TPM/TEE) is the only acceptable
  target for v2.

## 1. Fragmented vault storage with composition ordering

### Construction

The vault becomes a directory of small fragments under `~/.dota/v2/`:

- **`boot.shard`** -- committed at vault creation. Holds a 32-byte
  commitment seed `boot_seed`, KDF parameters, both public keys, the
  suite tag, and a `min_version` floor. *No private key material, no
  ciphertext.* This is the only shard with a deterministic name; an
  attacker who finds it learns nothing beyond what `dota info`
  already exposes.
- **`keys.shard`** -- holds the AES-GCM ciphertexts for the ML-KEM-768
  and X25519 private keys plus the v7 TC-HKEM key commitment.
- **N content shards (`<hex64>.shard`)** -- each holds M encrypted
  secrets (M small, e.g. 16/shard, so a 1000-secret vault has ~63
  content shards). Filenames are 32-byte BLAKE3 pseudonyms, not
  sequential indices; a directory listing leaks shard count but not
  which order they fit in or which holds which secret name.

### Composition chain (the "door knock")

Two separate derivations -- one stable across unlocks (so the shard
files we wrote yesterday can still be found today), one fresh per
unlock (so the order we touch them in does not leak a stable I/O
pattern):

```
mk           = Argon2id(passphrase, boot.shard.salt, ...)             # boot-bound, immutable for vault life
addr_seed    = HKDF(mk, "dota-v2-addr-seed", boot.shard.boot_seed)    # deterministic across unlocks
addr[i]      = BLAKE3(addr_seed, i.to_be_bytes())                     # filename of shard i: <hex64>.shard

session_id   = OsRng::next()                                          # fresh per unlock, never on disk
walk_seed    = HKDF(mk, "dota-v2-walk-seed", session_id)
walk[j]      = BLAKE3(walk_seed, j.to_be_bytes()) mod N               # permutation over [0, N), in-memory only
```

`addr[i]` is the *storage address*: shard i sits at `<addr[i]>.shard`
forever, so a fresh unlock can still find it. `walk[j]` is the
*traversal order* the current session uses to read or write shards:
permuted per-session so two consecutive unlocks of the same vault
touch the same files in different orders. The temporal binding is
`(boot_seed, session_id)` -- the first immutable per vault, the
second per session.

Per-shard:

```
shard_key_i  = HKDF(mk, "dota-v2-shard", addr[i] || boot.shard.suite)
plaintext_i  = AES-GCM-Decrypt(shard_key_i, shard_at(addr[i]).ciphertext)
```

An attacker without `mk` cannot compute `addr_seed`, cannot map
filenames back to shard indices, and so a complete dump of every
`.shard` file is useless without the passphrase. The fresh `walk[]`
defeats traffic-analysis pattern fingerprinting across unlocks; it
does not by itself prevent snapshot replay (see below).

### Does the fragmentation upgrade security? Honest assessment.

**Yes, conditionally.** Real improvements on three axes:

- **Single-fragment compromise** reveals only an AES-GCM blob plus a
  BLAKE3-derived filename. Without `mk`, the filename does not tell
  the attacker which secrets are in the shard.
- **I/O-pattern unlinkability across unlocks**: the per-session
  `walk[]` permutes the read/write order. Two consecutive unlocks of
  the same unchanged vault produce different I/O traces, defeating
  pattern fingerprinting on a system where an observer can correlate
  syscall traces with operations.
- **Partial restoration is impossible**: every legitimate unlock walks
  every `addr[i]`; a missing shard fails the integrity chain. No
  subset-of-secrets downgrade.

**What this construction does NOT provide:**

- **Anti-rollback (snapshot replay).** An attacker who can capture the
  whole `~/.dota/v2/` directory at time T and replay it at time T+N
  succeeds, because `addr[i]` is deterministic and the per-session
  `session_id` is generated locally and not bound to any monotonic
  state. Anti-rollback requires an explicit hardware-rooted counter:
  a TPM 2.0 NVRAM monotonic counter that increments on every vault
  write and is verified against the value recorded in `boot.shard` on
  every unlock. This is a v2.0 deliverable -- see section 3.

**Caveats** that constrain the upgrade:

- `boot.shard.boot_seed` is stored on disk. If the attacker reads it,
  they recover the per-vault half of the temporal binding. Mitigation:
  seal `boot.shard` to a TPM PCR policy (section 3) so it only
  unseals under measured boot.
- Filesystem metadata (entry count, file sizes, mtime) still leaks
  "how many shards exist" and "when was each last written."
  Mitigation: uniform-sized shards plus mtime randomization on every
  save.
- I/O pattern is observable. Every operation reads/writes shards.
  Mitigation: pad each operation to a fixed shard-touch count so
  `set foo` is indistinguishable from `get bar` at the FS-trace
  level.

The construction is sound. It is **not** sufficient on its own -- it
must compose with TPM sealing (section 3) and uniform-I/O padding to
deliver the full guarantee.

### Migration from v1.x

`dota upgrade --to v2`:

1. Unlock the v1.x vault via the existing v7 path.
2. Generate `boot_seed`, write `boot.shard`.
3. Wrap the existing private keys under `keys.shard`.
4. Split secrets into content shards of <= 16 each, written under their
   `addr[i]` pseudonyms.
5. Atomic directory swap (sibling staging dir renamed into place; fall
   back to a tombstone of the original on failure).

`migrated_from` in `boot.shard` carries the source version chain for
forensics, same shape as v1.x `MigrationInfo`.

### 2.2 Substantive `generate_salt` documentation

Current `src/crypto/kdf.rs:49-54` is a single line. Replace with:

```rust
/// Generate a 32-byte salt from the OS CSPRNG.
///
/// 32 bytes is the RFC 9106 archival recommendation for Argon2id and
```

Restores the *why* without naming an audit finding ID, naming a

### 2.3 Allowlist serde across every vault struct

`src/vault/format.rs` (Vault, KdfParams, KemKeyPair, X25519KeyPair,

Fix: add `#[serde(deny_unknown_fields)]` to every deserialized struct.
Implicit denylist -> explicit allowlist.

Trade-off: legacy migration becomes stricter. A vault with a future-
added field cannot be read by v2, which is the intended behavior
(`min_version` rollback floor was designed for this).

Same treatment for the new shard structs in section 1.

### 2.4 Side-channel posture in `//!` headers

Each crypto module gets a posture line stating what it relies on
upstream and what is out of the documented threat model. Factual,
not attacker-facing, no finding IDs.

Example for `src/crypto/x25519.rs`:

```
//! X25519 ECDH (via `curve25519-dalek`).
//!
//! Side-channel posture: scalar multiplication is constant-time per
//! upstream's documented invariants for the `static_secrets` feature
//! we enable. Frequency-channel attacks (Hertzbleed) and Apple
//! Silicon DMP-class attacks (GoFetch) are out of the documented
//! threat model for the single-process binary; the v2 mediated-trust
//! direction addresses them at a higher layer (TEE isolation), not
//! in this code.
```

Equivalent paragraphs for `crypto/aes_gcm.rs`, `crypto/mlkem.rs`,
`crypto/hybrid.rs`, `crypto/kdf.rs`, `crypto/legacy_kyber.rs`,
`security.rs`, `vault/ops.rs`. Each names only what upstream provides
and what higher layers are responsible for.

### 2.5 Firm allow-sets on validation

`validate_kdf_params` and `validate_v7_vault` are already allowlist-
shaped. The serde change in 2.3 closes the parallel channel where
unknown JSON fields slipped past them. Verify after the serde change:

- `validate_v7_vault` still asserts exact-equality for KEM and X25519
  algorithm strings.
- Every length field in `KemKeyPair`, `X25519KeyPair`,
  `EncryptedSecret` is asserted to be the exact expected byte length.
- New shard structs get the same exact-length treatment from day one.

## 3. TPM 2.0 sealing for the boot commitment

The fragmentation in section 1 needs the `boot.shard` to be readable
only under the same boot chain that created the vault. Otherwise a
same-UID attacker who copies `~/.dota/v2/` to another machine can
mount it as their own.

Mitigation: seal `boot.shard.boot_seed` to a TPM 2.0 policy bound to
PCRs 0, 2, 4, 7 (firmware, option ROM, bootloader, secure-boot key
hash). Seal at `dota init`. Unseal at every unlock. PCR mismatch
(re-image, different signing key, firmware update without re-seal)
fails the unseal and refuses to open the vault.

Implementation: `tss-esapi` crate (Rust binding around `tpm2-tss`).
Gated behind a `tpm` cargo feature so machines without a TPM still
run dota under a documented weaker posture.

First-class commands:
- `dota tpm-reseal` -- re-seal after a legitimate firmware update.
- `dota init --escrow-to <path>` -- write a TPM-decryptable copy of
  `boot.shard` to removable media, for motherboard-failure recovery.

This delivers "pre-boot chain of trust" without writing a UEFI app.

### Anti-rollback monotonic counter

Section 1's "What this construction does NOT provide" calls out that
the fragmentation alone does not stop snapshot replay: an attacker
who copies the whole vault directory at time T and restores it at
T+N succeeds. Closing that gap requires a hardware-rooted counter,
which the same TPM provides for free:

- At `dota init`, allocate a TPM 2.0 NVRAM index with attributes
  `NV_COUNTER | OWNERWRITE` and a write-policy bound to the same
  PCR set as the seal. Record the index handle in `boot.shard`.
- On every write transaction (set, rm, change-passphrase, rotate,
  tombstone hygiene), the daemon-side write path does:
  1. Read the current counter value `c0`.
  2. Stage the new vault state to a sibling directory.
  3. Increment the counter (`NV_Increment`), getting `c1 = c0 + 1`.
  4. Embed `c1` into the staged `boot.shard` (an authenticated
     field inside the sealed payload, so it travels with the
     resealed blob).
  5. Atomic rename the sibling directory into place.
- On unlock, the daemon reads `c_disk` from `boot.shard` and
  `c_tpm` from the NVRAM counter. `c_disk == c_tpm` proves the
  vault is the current generation. `c_disk < c_tpm` proves
  rollback (the disk content predates a write the TPM has
  recorded) and aborts the unlock.

This gives genuine anti-rollback: a snapshot taken at any past
generation fails the equality check against a counter the TPM
will not let the attacker decrement. The cost is one NVRAM index
per vault and one TPM round-trip per write.

`dota tpm-reseal` also resets the counter binding when a firmware
update legitimately invalidates the PCR seal; the previous counter
value is carried forward so legitimate state is preserved.

## 4. Mediated daemon -- research target, not v2.0 deliverable

The user rejected UDS, systemd socket activation, and D-Bus as
insufficient. They are correct: each runs the IPC through the
kernel's socket buffer, where root or a same-UID attacker with
`ptrace` capability can read it. v1.x's `harden_process` plus
`PR_SET_DUMPABLE = 0` raises the same-UID bar but not the root bar.

The genuine solution is hardware-rooted mediation: the secret-handling
process runs in a TEE (Intel SGX enclave, AMD SEV-SNP guest, Apple
Secure Enclave, ARM TrustZone) and the client communicates with it
via the platform's attested channel. The kernel never sees plaintext.

v2.0 scope is **research**: define the threat model, survey
available TEE platforms, document why each does or does not fit
(SGX deprecated by Intel, SEV-SNP requires a hypervisor and is
server-targeted, Secure Enclave is Apple-only and not directly
programmable for user apps, TrustZone is mobile-targeted). The
deliverable is a `dotav7-paper`-style technical note, not running
code.

A pragmatic intermediate (between "no daemon" and "TEE daemon") would
be a privileged-process model with hardware-attested IPC: dedicated
`dota` user, UDS at `/run/dota/dotad.sock`, every IPC message
end-to-end encrypted under keys derived from the TPM-sealed
`boot_seed`. The kernel sees ciphertext on the socket; root with
ptrace can still extract the daemon's session keys, but only by
attaching to a process that has authenticated. This does not match
the user's rejection literally; recommend re-evaluation after section
4's technical note lands.

## 5. UEFI / DRTM research stream

UEFI itself has a multi-year backlog of severe vulnerabilities
(LogoFAIL 2023, BootHole 2020, BlackLotus 2023, ESPecter 2021, the
ongoing Intel BootGuard bypass series). Writing a UEFI app without a
corresponding hardware partnership puts the vault's trust root on a
moving target.

v2.0 captures the outline only:

- Survey alternatives to "write a UEFI app": Intel TXT / AMD SKINIT
  (DRTM), measured boot via systemd-cryptenroll with TPM 2.0 sealing
  (which is what section 3 already delivers), MOK-shim mediation,
  Trenchboot late-launch.
- Identify the smallest UEFI surface that would actually move the
  needle: a DXE driver doing TPM PCR extends and measured-boot
  enforcement, chaining to the kernel. Smaller than "spawn the
  daemon from UEFI"; partially achieved already by systemd-boot
  with PCR extensions.
- Honest scoping: a UEFI deliverable requires firmware-signing
  infrastructure, OEM cooperation for production measured boot, and
  a hardware test matrix. Not in scope for v2.0. The plan documents
  the path; the work is post-v2.

## 8. User-injected variability layer

### Why this layer exists

Sections 1 through 7 specify cryptographic primitives and storage
mechanics. Any sufficiently capable adversary (including an LLM-driven
one that reads the source) can fully model that specification. The
primitives stay strong against a polynomial adversary, but the
deployment is *mappable*: an attacker who knows the design knows where
to look.

This section adds a layer whose specific parameterization is chosen by
the user at deploy time and never appears in source. The mechanisms
below do not add cryptographic strength; they add **deployment
unpredictability** so that an attacker who has read every line of dota
still does not know what *this particular vault* actually looks like.

The companion insight: bound the adversary's strategy space by making
the game's termination structure unknown to them. An attacker who
cannot precompute "I get N attempts in T hours, against M shards of
which K are real" cannot run a fully-optimized strategy. They have to
hedge. Hedging costs them time.

All mechanisms in this section are **opt-in**. The default vault is
the section-1-through-7 design. Users who explicitly enable a
variability feature pay the UX cost in exchange for moving-target
behavior.

### 8.1 Decoy shards

Vault directory contains M real shards and K decoys. Decoys are
indistinguishable from real shards: same byte length, same encryption
shape, AES-GCM-sealed under throwaway keys, decrypt to plausible noise
under the user's `mk`. M and K are chosen at init and recorded only
inside the TPM-sealed `boot.shard`. An attacker who compromises one
shard has probability `M/(M+K)` that it is real and cannot compute
either M or K without first unsealing `boot.shard`.

### 8.2 Knock pattern

User-defined sequence of (operation, delay, optional input) recorded
at init. Required for unlock *in addition* to the passphrase. The
sequence itself is stored as a Merkle commitment in `boot.shard`; the
plaintext sequence lives only in the user's memory (or on a separate
ritual-recovery medium, see "Recovery" below).

The choreography raises the unlock cost from "guess the passphrase" to
"guess the passphrase AND reproduce a multi-step interaction the user
chose at init." Adversaries who scrape known-passphrase corpora and
replay them against dota cannot succeed without also discovering the
knock.

### 8.3 Honey-token re-shard trigger

A subset of the decoys are tripwires. Touching a tripwire shard (read
or write) triggers re-derivation of `addr_seed` on the next legitimate
save: every real shard moves to a new filename. An attacker who picks
the wrong shell has actively destroyed their reconnaissance map and
gains nothing by trying again -- the map they built has been replaced.

The defender does nothing manual; the next legitimate write rebuilds
the directory under fresh addresses.

### 8.4 Duress passphrase + honey-vault

A secondary passphrase opens a separate vault filled with user-
supplied plausible-fake secrets. Coerced unlock yields data. The
coercer cannot distinguish the honey-vault from the real one without
external verification of the contents.

Standard plausible-deniability primitive (VeraCrypt hidden volumes is
the closest prior art). Has the same well-known UX failure modes; the
defaults stay off.

### 8.5 Variable-time unlock backoff

Failed unlock attempts trigger a backoff drawn from a user-chosen
distribution (linear, exponential, or random pause within bounds).
Distribution parameters set at init, stored in the TPM-sealed
`boot.shard`. The attacker cannot precompute "I get N attempts in T
hours" because they do not know the curve.

### 8.6 Cipher-suite selection from a permitted set

At init the user selects one of N approved primitive combinations
(e.g., `ML-KEM-768 + X25519 + AES-256-GCM` versus
`ML-KEM-1024 + Ed448 + ChaCha20-Poly1305`). The choice is per-vault,
stored in `boot.shard`. An attacker has to compromise `boot.shard`
before knowing which crypto is actually in use, which forecloses
preparing a primitive-specific attack against the vault directory in
advance.

### Recovery is the hard secondary problem

If the user forgets the knock pattern or the decoy count or the
duress passphrase, the vault is permanently sealed even from them.
This is the 20-year recurring failure mode of plausible-deniability
systems.

Mitigations:

- **Ritual recovery medium**: a separate object (paper, hardware
  key, second TPM-sealed file on an air-gapped USB) holds the
  choreography encrypted under a recovery passphrase. The recovery
  medium is required to be physically separate from the vault.
- **All variability features default off.** Users opt in
  deliberately; defaults are the simpler section-1-through-7 design.
- **`dota knock-test`** dry-run command lets a user verify their
  choreography works against a throwaway vault before committing to
  one that holds real secrets.
- **`dota init --variability=full --dry-run`** at init time prints
  the recovery medium contents to stdout for the user to record on a
  separate device before the vault is sealed.

### What this layer does NOT provide

- It does not protect against a fully compromised host (the
  attacker has the choreography by definition once the host is
  compromised).
- It does not protect against a user whose passphrase is also their
  knock pattern. Some users will do this; the construction cannot
  stop them.
- It does not increase the cryptographic strength of the primitives
  in section 1; it operates entirely at the protocol layer.
- It adds significant UX cost. The defaults are not the variability-
  enabled mode for exactly that reason.

The goal of v1.1.0 is to close every remaining finding, add regression coverage for each fix, and ship a single audit-cleanup PR.

Three guardrails learned the hard way (commit history of PR #16) gate every change in this plan:

1. **Read upstream Cargo.toml feature flags before relying on Error trait impls.** `hkdf` 0.12 and `hmac` 0.12 ship with `default = []`; their `InvalidPrkLength` / `digest::InvalidLength` only implement `std::error::Error` when the `std` feature is enabled. The H2 commit chain re-learned this twice. Decision: **do not enable the `std` feature on these crates during v1.1.0** -- leave the existing `map_err(|e| anyhow!("...: {}", e))` lines at `vault/ops.rs:889-898, 935-937, 964-967` in place (and documented in code) and defer the cleanup to v1.2.0 with its own validation cycle.
2. **Don't push speculative crypto-adjacent changes without exhausting local validation.** Every crypto-adjacent commit (M2 okm, H3 tombstone, H4 ml-kem upgrade) passes `cargo fmt --check`, `cargo clippy --all-targets --all-features -- -D warnings`, `cargo clippy --all-targets --no-default-features -- -D warnings`, `cargo test --all-features`, `cargo test --no-default-features`, `cargo build --release`, **and** the manual end-to-end CLI sweep in S4 before commit. No exceptions.
3. **Weigh README-precision findings separately from security-posture findings.** H2's resolution treated the readme drift as a docs defect and the `.expect` calls as a (separate) code defect. Same separation applies to M4/M9/L1-L7 (docs/style) vs H1/H3/H4/M1/M2/M6/M7/M8 (posture).

- **Release shape**: v1.1.0 minor, single PR on `claude/fix-security-tests-kH8TP`.
- **H4**: upgrade ml-kem to latest **stable 0.3.2** (verified via `cargo info ml-kem`), pin with `=0.3.2`, feature-gate `pqcrypto-kyber` behind a new `legacy-migration` feature, drop dead deps. Validate full crypto path end-to-end before commit.
